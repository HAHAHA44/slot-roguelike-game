# Slot Roguelike PRD

## 文档信息
- 项目代号：`Project Reelbound`（暂定）
- 文档类型：Product Requirements Document
- 版本：`v0.1`
- 日期：`2026-03-24`
- 状态：Draft
- 适用阶段：原型验证 -> MVP -> EA 扩展

## 1. 产品概述
本项目是一款基于 5x5 slot grid 的轻度 Roguelike 构筑游戏。玩家在有限轮次内，通过放置、删除、替换与联动 Token，在周期性结算中达成分数目标并持续晋级。核心体验参考《幸运房东》的“盘面构筑 + 周期压力”结构，但产品目标不是单纯复制，而是在不改变基础规则的前提下，围绕事件系统、角色系统、局外成长和可扩展空间规则，建立更强的 build 感、更明确的爽点、以及更长期的内容扩展能力。

本 PRD 锁定如下基础玩法，不允许被后续设计改写：
- 固定基础棋盘为 `5x5 slot grid`
- 玩家通过放置 Token 形成相邻 / 行列 / 条件触发
- 每 `N` 轮进行一次结算，不达标即失败
- 每轮提供三个基础操作入口：新 Token、删除 Token、随机 Token
- Token 具备稀有度与类型
- 多轮达标后通关，通关后进入 Endless 模式

## 2. 设计目标
### 2.1 产品目标
- 单局时长稳定在 `10~25 分钟`
- 玩家每局都能形成清晰的构筑方向，而不是只做局部优化
- 局内存在明显爆发回合和连锁反馈
- 玩家持续面临“赌当前收益”与“保后期稳定”的风险决策
- 内容结构必须数据化，适合后续持续扩充 Token、事件、角色、难度和主题皮肤

### 2.2 设计支柱
1. 盘面本身就是构筑
2. 事件系统驱动转型，而不是只做随机打断
3. 角色提供局内倾向，不替代盘面思考
4. 局外成长提供解锁与选择广度，不直接碾压局内平衡
5. 扩展规则必须“低频、高光、可控”，避免基础系统被稀释

## 3. 目标用户与产品定位
### 3.1 核心用户
- 喜欢《幸运房东》《Balatro》《Loop Hero》式轻 Roguelike 构筑的玩家
- 偏好短局、多次重开、理解联动后获得爆发快感的玩家
- 接受一定随机性，但希望有明确策略空间的独立游戏用户

### 3.2 平台与商业假设
- 首发平台：PC
- 控制方式：鼠标优先
- 商业模式：一次性买断制优先
- 美术方向：主题中性，支持后续换皮
- 说明：若后续平台、商业模式改变，本 PRD 的系统要求基本不变

## 4. 成功标准
### 4.1 体验成功标准
- 玩家在前 2 局内能理解结算压力与基础联动逻辑
- 玩家在前 5 局内至少体验到一次明显“连锁爆发”时刻
- 玩家在失败后能够明确归因于构筑选择，而非纯随机背刺
- 玩家在首次通关前，能识别至少 3 条不同构筑方向

### 4.2 系统成功标准
- MVP 至少支持 `4` 条可成型构筑方向
- MVP 至少支持 `3` 类事件类型：即时、持续、危机
- MVP 至少支持 `3` 名角色原型
- MVP 至少支持 `6` 个 Ascension 难度修正项中的前 `3` 层
- 内容配置支持后续通过数据表扩充，不依赖重写结算框架

## 5. 核心玩法循环
### 5.1 单轮循环
1. 玩家进入回合
2. 执行转动 / 触发 / 收集结算
3. 更新 Token 状态（充能、倒计时、临时效果等）
4. 给出回合奖励选择：
   - 新 Token
   - 删除 Token
   - 随机 Token
5. 若达到特殊节点，则触发事件选择或系统奖励
6. 进入下一轮

### 5.2 周期结算循环
1. 每 `N` 轮触发一次目标分数检查
2. 达标则进入下一阶段，并可获得强化机会
3. 未达标则本局失败
4. 多次达标后完成标准通关
5. 通关后进入 Endless 模式，继续使用相同盘面挑战无限压力增长

### 5.3 决策层级
- 短期：当前回合选择哪个 Token / 是否删除 / 是否赌随机
- 中期：本阶段围绕哪个事件方向构筑
- 长期：是否为后续 Boss 结算、稀有事件、空间异常保留布局

## 6. 不可变基础规则
本节为硬约束，后续设计只能在其外侧扩展：
- 基础棋盘默认为 `5x5`
- Token 的主要交互来自：
  - 相邻
  - 行列
  - 条件判断
- 分数压力以固定周期结算为核心，不改为实时血量制或纯关卡制
- 每轮必须保留“新增 / 删除 / 随机”三选项的决策框架
- 通关后必须有 Endless 模式

## 7. 核心系统需求
### 7.1 Token 系统
#### 7.1.1 Token 基础字段
每个 Token 必须至少包含：
- `id`
- `name`
- `rarity`
- `type`
- `tags`
- `base_value`
- `trigger_rules`
- `state_fields`
- `spawn_rules`
- `remove_rules`

#### 7.1.2 稀有度设计
- Common：基础联动件、低门槛过渡件
- Uncommon：构筑方向件、触发放大件
- Rare：流派核心件、系统扭曲件
- Legendary：高光件、空间/规则突破件

#### 7.1.3 标签系统
为确保主题可替换和内容可扩展，Token 主要通过抽象标签驱动，而不是写死题材语义。MVP 标签建议包含：
- `Grow`
- `Break`
- `Link`
- `Charge`
- `Guard`
- `Wild`
- `Spatial`

标签的目标：
- 让事件系统可统一读取盘面
- 让角色技能可影响某类构筑而不是写死单一 Token
- 让 AI 后续扩内容时遵循相同语义体系

#### 7.1.4 Token 设计要求
- 每个 Token 至少服务于一个明确流派
- 稀有 Token 不允许只是“数值更大”
- 每个流派至少有：
  - 早期过渡件
  - 中期放大件
  - 后期终局件
- 删除和随机获得必须可成为正向策略，而非纯惩罚或纯赌博

### 7.2 结算与触发系统
#### 7.2.1 触发类型
- 相邻触发：直接四向或八向，MVP 推荐四向以控制复杂度
- 行列触发：按所在行、列或整条覆盖关系判定
- 条件触发：满足状态、标签数量、删除、转化、倒计时等条件

#### 7.2.2 结算顺序要求
结算顺序必须固定，并可视化，避免玩家无法推导结果。建议顺序：
1. 基础产出
2. 相邻触发
3. 行列触发
4. 条件触发
5. 复制/放大类效果
6. 临时物消失与收尾结算

#### 7.2.3 爆发回合要求
- 每局应至少出现 1 次可感知的爆发回合
- 爆发需要有铺垫，不允许多数强力件无条件稳定产出
- 大额收益最好通过“前置充能 / 链式复制 / 限时契约”实现

### 7.3 事件系统
事件系统是本产品的第一扩展核心，必须承担“塑造构筑方向”和“制造中局转型”的职责。

#### 7.3.1 事件分类
- 即时事件：立即生效，改变当前盘面或资源
- 持续事件：在若干轮内持续生效，鼓励围绕其构筑
- 危机事件：提供高压目标与高额奖励，失败有明确惩罚
- 异常事件：极稀有，改写空间或规则

#### 7.3.2 Event Draft 机制
- 在关键结算节点后，从 `3` 个事件中选择 `1` 个
- 事件池权重由盘面标签、角色倾向、难度层级共同影响
- 至少提供一个相对稳健选项，避免纯死局

#### 7.3.3 风险契约机制
- 玩家可在事件中签订附加目标
- 典型结构：`时限 + 目标 + 奖励 + 失败惩罚`
- 奖励应以方向性资源为主，例如：
  - 指定标签 Token 刷新机会
  - 免费删除次数
  - 稀有事件权重提升
  - 临时倍率
- 惩罚应避免直接结束本局，优先采用：
  - 随机删除
  - 下轮事件质量下降
  - 暂时封锁一个选择栏位

#### 7.3.4 异常事件机制
用于承接未来“打破基础玩法”的高光内容，包括但不限于：
- 临时扩展到 `6x6`
- 启用一列临时格位
- 生成占 `2` 格或更大面积的 Token
- 反转部分触发逻辑
- 引入短期天气/场景规则

要求：
- 低频出现
- 必须在 UI 上强提示
- 对基础平衡的影响可控
- 不作为通关必要条件

### 7.4 主角系统
主角系统用于建立开局身份差异和局内倾向，不应碾压 Token 构筑本身。

#### 7.4.1 主角基础结构
每个主角包含：
- 固有属性倾向
- 初始被动
- 可解锁技能树或天赋节点
- 专属事件权重修正

#### 7.4.2 建议属性
- `Insight`：提升事件可见数、信息质量、规划能力
- `Resolve`：降低危机失败惩罚、提高保底稳定性
- `Flux`：提高随机收益和异常事件概率
- `Greed`：提高高风险高回报选项收益，同时增加坏结果波动

#### 7.4.3 主角技能原则
- 技能应改变决策偏好，而非直接送钱
- 技能效果优先影响：
  - 事件权重
  - 标签协同
  - 删除/随机选项质量
  - 临时规则持续时间
- 避免“开局送核心神件”型设计

### 7.5 局外成长（Meta Progression）
局外成长的目标是扩展可玩性与选择广度，而不是单纯叠数值。

#### 7.5.1 局外成长内容
- 新角色解锁
- 新事件池解锁
- 新 Token 池解锁
- 新难度层解锁
- 新异常规则解锁
- 新起始 relic/徽记解锁（可选）

#### 7.5.2 局外资源原则
- 通关、失败、完成挑战均应提供少量局外资源
- 首通与高难挑战给额外奖励
- 局外成长应更多解锁“可能性”，少做永久伤害/收益加成

#### 7.5.3 禁止项
- 不允许出现“没刷够局外数值就无法通关普通难度”的设计
- 不允许核心功能被藏在过深局外成长后

### 7.6 难度系统（Ascension）
难度系统用于延长生命周期和验证构筑深度。

#### 7.6.1 难度修正来源
- 提高目标分数曲线
- 减少容错轮数
- 提升危机事件出现率
- 降低事件 Draft 质量
- 增加随机负面干扰
- 削弱保底选项

#### 7.6.2 Ascension 设计原则
- 每层难度只引入 1~2 个明确规则变化
- 难度提升应优先增加决策压力，不优先堆纯数值
- 至少保留一种保守玩法能够稳定冲层
- 高风险玩法在高 Ascension 仍应有理论天花板优势

### 7.7 Endless 模式
- 标准通关后解锁
- 使用同一盘面继续游玩
- 难度以可预期曲线持续抬升
- 主要用途：
  - 满足高构筑玩家的“继续爽”
  - 验证后期构筑强度
  - 形成截图/传播内容

## 8. 可扩展主题设计
本项目要求主题模糊化、语义抽象化，确保系统能被不同题材包装。

### 8.1 主题中性设计原则
- 系统语义优先使用标签和功能，而不是强题材命名
- 美术、文案、音效可替换，但不改变数据结构
- Token 设计先定义“玩法功能位”，再决定外观皮肤

### 8.2 内容层拆分
建议将内容拆为三层：
1. 系统层：稀有度、标签、触发规则、状态机
2. 题材层：名称、图标、文案、音效、动画
3. 扩展层：事件包、角色包、主题皮肤包

### 8.3 AI 内容扩展要求
后续 AI 生成 Token/事件时，必须遵循固定模板：
- 功能定位
- 标签
- 触发条件
- 输出类型
- 风险等级
- 是否会形成复制循环
- 是否会制造死局

## 9. 打破基础玩法的扩展系统
该系统不是开局默认能力，而是中后期稀有内容入口。

### 9.1 扩展目标
- 为长期版本提供高光差异
- 让高稀有内容拥有真正“改规则”的惊喜感
- 保证基础 5x5 依然是主舞台

### 9.2 可扩展方向
- `6x6` 临时扩盘
- 多格 Token
- 临时障碍格 / 增幅格 / 传导格
- 事件改写结算顺序
- 某阶段允许 Token 合成或裂变

### 9.3 系统约束
- 扩展规则必须是模块化开关
- 每次只同时激活少量空间异常
- 扩展内容必须在数据层声明占格形状、旋转规则、结算优先级

## 10. 样例内容定义
### 10.1 首轮事件机制并入产品需求
以下设计并入正式需求：

#### Event Draft
- 结算后 3 选 1 事件
- 事件由盘面标签、主角属性、当前阶段共同加权
- 目标：把中盘从“补数字”转变为“转方向”

#### Tag Pulse
- 所有 Token 使用统一标签系统
- 事件与角色技能都通过标签读取盘面状态
- 目标：为内容扩展提供统一语言

#### Risk Contract
- 玩家主动签订限时条件换取方向性奖励
- 目标：制造赌 vs 稳的博弈

#### Spatial Anomaly
- 极稀有事件提供规则突破
- 目标：预埋高版本扩展空间

### 10.2 样例 Token 池
#### Pulse Seed
- 稀有度：Common
- 标签：`Grow` `Charge`
- 效果：相邻有 `Grow` 或 `Charge` 时获得充能；满层后本次结算翻倍
- 用途：基础蓄爆件

#### Relay Prism
- 稀有度：Uncommon
- 标签：`Link`
- 效果：复制本行第一次触发的相邻效果到同列随机目标
- 用途：连锁放大件

#### Hollow Shell
- 稀有度：Common
- 标签：`Break`
- 效果：删除时产生临时碎片；碎片消失时若满足条件则得分
- 用途：删除收益件

#### Anchor Glyph
- 稀有度：Rare
- 标签：`Guard`
- 效果：保护相邻单位免受随机破坏，并在触发时获得有限永久成长
- 用途：保底稳定件

#### Wild Signal
- 稀有度：Rare
- 标签：`Wild`
- 效果：临时复制周边多数标签并追加一次该标签计数
- 用途：混合流派补足件

#### Twin Monolith
- 稀有度：Legendary
- 标签：`Spatial` `Link`
- 效果：占两格并按两端邻接分别触发
- 用途：空间扩展原型件

## 11. 用户体验与反馈需求
### 11.1 爽点设计要求
- 高频小反馈：普通触发也要有清晰跳字、连线、音效
- 中频构筑反馈：拿到方向型事件或核心件时有明显提示
- 低频大高潮：爆发回合、契约完成、异常事件触发必须有强烈视听反馈

### 11.2 挫败控制
- 玩家失败时必须能回看关键原因
- 事件失败惩罚应可理解、可预测
- 随机破坏类效果需要预告与缓冲，不做完全无提示背刺

### 11.3 信息可读性
- 每个 Token 必须显示：
  - 当前状态
  - 触发来源
  - 本轮贡献分
- 事件界面必须显示：
  - 生效轮数
  - 契约条件
  - 失败惩罚
  - 关键受影响标签

## 12. 数值与平衡要求
### 12.1 必须重点监控的风险
- 复制触发闭环
- 永久成长叠加失控
- 多格 Token 无脑成为最优解
- 危机事件导致无解负面局
- Wildcard 类 Token 成为所有流派通吃件

### 12.2 限制原则
- 复制类效果设置每轮上限
- 永久成长设置硬上限或递减收益
- 危机事件至少提供一个低收益保底路线
- 异常事件不作为唯一通关解
- 多格 Token 数量与出现率严格受限

### 12.3 分数曲线要求
- 前期给玩家快速建立联动的正反馈
- 中期提高方向选择成本
- 后期要求围绕核心机制形成爆发，否则无法稳定达标

## 13. MVP 范围
### 13.1 必做项
- 基础 5x5 盘面与固定结算循环
- 基础 Token 放置 / 删除 / 随机获取
- 标签化 Token 系统
- Event Draft
- 即时事件 / 持续事件 / 危机事件
- 3 名主角原型
- 基础局外解锁
- Ascension 前 3 层
- 标准通关 + Endless 模式

### 13.2 推荐首批内容量
- Token：`45~60`
- 事件：`20~30`
- 主角：`3`
- 局外解锁节点：`15~25`
- 异常事件：`2~4`

### 13.3 暂不进入 MVP 的内容
- 大量剧情文本
- 大规模主题皮肤系统
- 复杂多阶段 Boss 战
- 超过 `2x1` 的复杂多格形状
- 在线功能与排行榜依赖玩法主循环

## 14. 内容生产与技术要求
### 14.1 数据驱动要求
以下内容必须可配置化：
- Token 定义
- 事件定义
- 角色定义
- 难度修正项
- 局外成长节点
- 空间异常规则

### 14.2 内容模板要求`n每个 Token/事件配置必须包含：
- 功能定位
- 稀有度
- 标签
- 触发时机
- 影响范围
- 可堆叠性
- 风险说明
- 是否可能破坏平衡

### 14.3 可测试性要求
- 结算顺序必须可复现
- 任一事件和 Token 组合必须能在日志中追溯
- 难度修正项必须支持单独开关测试
- 多格 Token 与扩盘事件必须支持独立模拟测试

## 18. 结论
本 PRD 保留了你定义的基础玩法，并将事件系统、角色系统、局外成长、难度系统和规则突破系统统一进一个可持续扩展的产品框架。MVP 的核心不是“内容堆量”，而是先验证三件事：事件是否真的能驱动构筑、角色是否真的改变打法、以及异常规则是否能在不破坏基础 5x5 的前提下提供高光差异。
